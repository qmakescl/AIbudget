## 2. 데이터 추출 v2 엔진 도입

### 2.1 v1의 문제점

기존 `extract_all.py`(v1)는 **PyMuPDF 텍스트 파싱을 1차**로 사용하고, pdfplumber를 fallback으로 운용했다. 이로 인해 다음 문제가 발생했다.

- **표 행 섞임**: PyMuPDF는 텍스트를 페이지 상 좌표 순서로 읽기 때문에, 한 페이지에 두 개의 표가 있을 경우 행이 뒤섞였다.
- **사업내용(description) 미추출**: 비정형 텍스트 영역에서 설명문을 구분하지 못해 추출률 0.6%에 불과했다.

### 2.2 v2 설계: pdfplumber 우선 + 3단계 fallback

`verify_extract.py`(v2)는 추출 우선순위를 역전시켰다.

```
1차: pdfplumber 표 추출      ← 셀 경계 보존, 표 혼합 없음
2차: PyMuPDF 텍스트 pass1   ← 60줄 범위, 조기 break
3차: PyMuPDF 텍스트 pass2   ← 80줄 범위, no-break (더 넓게 탐색)
교차 검증: 두 방법 결과 비교  ← 불일치 시 플래그 처리
```

### 2.3 사업내용(description) 추출 개선

| 구분 | 추출 성공 건수 | 비율 |
|------|--------------|------|
| v1 | ~3건 | 0.6% |
| v2 | 531건 | 99.6% |

pdfplumber로 표를 추출한 뒤 표 영역 **이후**의 텍스트 블록을 별도로 읽어 description 필드에 매핑하는 방식으로 해결했다.

## 3. 대시보드 데이터 스키마 마이그레이션

### 3.1 스키마 변경 내용

`prepare_data.py`가 생성하는 `data.json` 구조가 전면 재설계되었다.

**이전 스키마 (v1)**
```json
{
  "summary": {
    "total_budget_2026": 27612345,
    "increase_projects": 416,
    "decrease_projects": 98
  },
  "dept_stats": [
    { "budget_2026": 1234567 }
  ]
}
```

**새 스키마 (v2)**
```json
{
  "kpi": {
    "total_budget_2026_bil": 27.61,
    "increased_projects": 416,
    "decreased_projects": 98,
    "frozen_projects": 3,
    "new_projects": 188
  },
  "meta": {
    "net_change_mil": 5944200
  },
  "dept_summary": [
    { "budget_2026_mil": 1234567 }
  ],
  "projects": [...]
}
```

주요 변경점:
- `data.summary` → `data.kpi` + `data.meta` (역할 분리)
- `data.dept_stats` → `data.dept_summary`
- `total_budget_2026` (백만원 raw) → `total_budget_2026_bil` (조원, 이미 변환됨)
- `increase_projects` → `increased_projects` (복수형 통일)
- `decrease_projects` → `decreased_projects`
- `d.budget_2026` → `d.budget_2026_mil` (단위 명시)

### 3.2 dashboard.js 수정 위치

| 함수 | 변경 내용 |
|------|----------|
| `initDashboard()` | `renderKPIs(data.summary)` → `renderKPIs(data.kpi, data.meta)` |
| `renderKPIs()` | 인자 2개로 분리, `kpi.total_budget_2026_bil.toFixed(2)` (단위 변환 불필요) |
| `renderKPIs()` | `meta.net_change_mil / 100` (백만원 → 억원) |
| `renderAllTab()` | `data.dept_stats` → `data.dept_summary`, `d.budget_2026` → `d.budget_2026_mil` |
| 도넛 차트 | `kpi.frozen_projects` 직접 사용 |

### 4.2 패키지 설치

기존 uv 가상환경(`.venv/`)에 필수 패키지를 설치했다.

```bash

```

### 4.3 pyproject.toml 업데이트

의존성을 명시적으로 선언했다.

```toml
[project]
dependencies = [
    "pandas>=2.0.0",
    "pdfplumber>=0.10.0",
    "pymupdf>=1.23.0",
]
```

### 4.4 uv.lock 생성

`uv sync` 실행으로 `uv.lock`(487줄)을 생성했다. 이를 통해 사용자가 `uv sync` 한 줄로 동일한 환경을 재현할 수 있다.

잠긴 주요 패키지:
- cffi, charset-normalizer, cryptography, numpy
- pandas, pdfminer-six, pdfplumber, pillow
- pycparser, pymupdf, pypdfium2, python-dateutil, six

| 제외 항목 | 이유 |
|----------|------|
| `.venv/` | 가상환경 (재현 가능) |

## 5. GitHub 브랜치 전략 및 배포

### 5.1 요구사항

- 기존 GitHub `main` 브랜치의 v1 코드를 보존
- 새 코드(v2)를 `main` 브랜치로 등록

### 5.2 실행 순서

```bash
# 1. 원격 저장소 연결
git remote add origin https://github.com/qmakescl/AIbudget.git

# 2. 원격 이력 가져오기
git fetch origin

# 3. 기존 main을 v1 브랜치로 보존
git branch v1 origin/main
git push origin v1

# 4. 새 코드를 main으로 강제 Push
git push --force origin main
```

> **중요**: v1 브랜치를 먼저 push한 뒤 main을 force push해야 이전 코드가 손실되지 않는다.

### 5.3 최종 브랜치 구조

| 브랜치 | 내용 |
|--------|------|
| `main` | v2 코드 (현재) — verify_extract.py, 개선된 dashboard, uv.lock, serve.py |
| `v1` | v1 코드 (보존) — extract_all.py 기반 원본 |

## 6. 로컬 서버 스크립트 추가

### 6.1 serve.py 생성

사용자가 한 줄 명령으로 대시보드를 실행할 수 있도록 `serve.py`를 작성했다.

```python
import http.server, os, sys, webbrowser
from pathlib import Path

def main():
    port = int(sys.argv[1]) if len(sys.argv) > 1 else 8000
    dashboard_dir = Path(__file__).parent / "dashboard"
    os.chdir(dashboard_dir)
    url = f"http://localhost:{port}"
    print(f"대시보드 서버 시작: {url}")
    webbrowser.open(url)
    handler = http.server.SimpleHTTPRequestHandler
    handler.log_message = lambda *args: None
    with http.server.HTTPServer(("", port), handler) as httpd:
        httpd.serve_forever()
```

### 6.2 실행 방법

```bash
uv run python serve.py        # 기본 포트 8000
uv run python serve.py 8080   # 포트 지정
```

브라우저 자동 오픈 + `dashboard/` 디렉토리 서빙.

### 6.3 GitHub Pages와의 호환성

`serve.py`는 저장소 루트(`/`)에 위치하며, GitHub Pages는 `dashboard/` 폴더만 서빙한다. 따라서 `serve.py`는 GitHub Pages에 아무 영향을 주지 않는다.
